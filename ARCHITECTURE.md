# Архитектура сервиса

## 1. Монолит или микросервисы

Для учебного проекта выбран монолитный подход: Flask API, загрузка модели и инференс находятся в одном сервисе.

Причины выбора:

- проект небольшой и решает одну задачу: принять признаки клиента и вернуть прогноз дефолта
- модель одна, отдельные сервисы для данных, модели и API пока усложнили бы поддержку
- монолит проще запускать локально и в Docker
- такой вариант достаточно похож на production-сервис, но остается понятным для учебного проекта

Микросервисный подход был бы полезен, если бы в системе появились разные модели, отдельная команда для ML-платформы, высокая нагрузка или независимое масштабирование частей системы.

## 2. Брокер сообщений

В текущей реализации брокер сообщений не используется, потому что `/predict` отвечает синхронно: клиент отправляет JSON и сразу получает прогноз.

В сценарии масштабирования можно добавить RabbitMQ или Kafka. Брокер пригодился бы для задач, которые не обязательно выполнять прямо во время HTTP-запроса:

- асинхронная обработка батч-предсказаний
- постановка тяжелых заявок в очередь
- отправка логов и событий в отдельный сервис аналитики
- повторная обработка сообщений при временных ошибках

Например, API мог бы принять файл с заявками, положить задачу в RabbitMQ, а отдельный worker считал бы сообщения из очереди и сохранял результаты в базу данных.

## 3. Логирование и мониторинг

Сервис пишет логи в stdout в JSON-формате. Это удобно для Docker: контейнерные логи можно смотреть командой `docker logs`, а в production их можно собирать централизованно.

Для `/health` логируется:

- событие `health_check`
- HTTP-метод
- путь
- статус ответа
- время обработки

Для `/predict` логируется:

- событие `predict_success` или `predict_error`
- HTTP-метод
- путь
- статус ответа
- время обработки
- версия модели
- прогноз
- вероятность дефолта

В production такие логи можно отправлять в ELK-стек, Grafana Loki или другой инструмент мониторинга. По ним можно отслеживать количество запросов, ошибки API, среднее время ответа и распределение прогнозов модели.

## 4. DVC и MLflow

DVC нужен для контроля версий данных и моделей. В этом проекте он мог бы хранить версии исходного датасета, промежуточных данных и файла обученной модели.

MLflow нужен для отслеживания экспериментов. В нем можно было бы сохранять параметры обучения, выбранный threshold, метрики модели и путь к сохраненному артефакту.

В текущей версии DVC и MLflow описаны концептуально, без реализации, так как согласно заданию достаточно кратко объяснить их назначение.

## 5. Бизнес-метрики

Помимо технических метрик модели, заказчику важны бизнес-метрики.

### Ожидаемые финансовые потери

Можно оценивать сумму потенциальных потерь по клиентам, для которых модель прогнозирует дефолт:

```text
expected_loss = probability * credit_limit * loss_rate
```

Где `probability` - вероятность дефолта из модели, `credit_limit` - кредитный лимит клиента, `loss_rate` - ожидаемая доля потерь при дефолте.

### Доля одобренных заявок при контроле риска

Можно сравнивать версии модели по тому, сколько клиентов получают одобрение при заданном максимальном уровне риска. Например, банк может хотеть увеличить долю одобренных заявок, не повышая ожидаемые потери.

Эта метрика помогает понять, дает ли новая модель бизнесу больше качественных клиентов, а не только лучшее значение F1-score.
