# План A/B-тестирования

## 1. Цель теста

Цель A/B-теста - сравнить текущую модель `v1` и новую модель `v2` для прогнозирования дефолта по кредитным картам.

Контрольная группа использует текущую модель `v1`.

Тестовая группа использует новую модель `v2`.

Главная задача - понять, дает ли новая модель лучшее качество определения дефолтных клиентов без ухудшения бизнес-показателей.

## 2. Разделение трафика

Трафик делится случайно в пропорции 50/50:

- 50% запросов идут в контрольную группу `v1`
- 50% запросов идут в тестовую группу `v2`

Разделение лучше делать по стабильному идентификатору клиента, например `client_id`. Так один и тот же клиент всегда будет попадать в одну и ту же группу.

Если `client_id` нет, можно использовать случайное распределение на уровне API, но для реального A/B-теста стабильное распределение надежнее.

## 3. Продолжительность теста

Минимальная продолжительность теста - 2 недели.

Такой срок нужен, чтобы собрать данные за разные дни недели и снизить влияние случайных колебаний в потоке заявок.

Если заявок мало, тест нужно продолжать дольше, пока в обеих группах не накопится достаточно наблюдений по классу дефолта.

## 4. Метрики

### Основная метрика

Основная метрика - `F1-score` для класса дефолта.

Эта метрика выбрана, потому что в задаче есть дисбаланс классов. Accuracy может быть высокой даже тогда, когда модель плохо находит дефолтных клиентов.

`F1-score` учитывает и precision, и recall, поэтому лучше подходит для сравнения моделей в этой задаче.

### Дополнительная метрика

Дополнительная метрика - `Recall` для класса дефолта.

Recall показывает, какую долю реальных дефолтов модель смогла найти. Для банка это важно, потому что пропущенный дефолт может привести к финансовым потерям.

При этом recall нельзя оценивать отдельно от precision, потому что слишком высокий recall может привести к большому количеству ложных отказов хорошим клиентам.

### Бизнес-метрики

Дополнительно можно смотреть:

- ожидаемые финансовые потери
- долю одобренных заявок при заданном уровне риска

Эти метрики описаны подробнее в [ARCHITECTURE.md](ARCHITECTURE.md).

## 5. Статистический анализ

Для сравнения основной метрики используется bootstrap.

Причина выбора: `F1-score` является составной метрикой, поэтому для него удобнее строить доверительный интервал через повторные выборки, а не через простой t-test.

План анализа:

- собрать результаты предсказаний для групп `v1` и `v2`
- для каждой группы много раз строить bootstrap-выборки
- на каждой выборке считать `F1-score`
- получить распределение разницы `F1_v2 - F1_v1`
- построить 95% доверительный интервал для этой разницы

Для дополнительной проверки recall можно использовать z-test для двух долей, потому что recall можно представить как долю найденных дефолтов среди всех реальных дефолтов.

## 6. Доверительные интервалы

Для `F1-score` доверительный интервал строится bootstrap-методом:

```text
delta = F1_v2 - F1_v1
```

После повторных выборок берутся 2.5% и 97.5% квантили распределения `delta`.

Так получается 95% доверительный интервал для разницы качества между моделями.

## 7. Критерий успеха

Тест считается успешным, если выполняются условия:

- `F1-score` модели `v2` выше, чем у `v1`
- 95% доверительный интервал для `F1_v2 - F1_v1` полностью выше 0
- recall модели `v2` не ниже recall модели `v1`
- бизнес-метрики не ухудшились

Если `v2` улучшает `F1-score`, но резко снижает recall или ухудшает бизнес-метрики, модель не стоит выкатывать полностью.
